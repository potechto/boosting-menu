2. PERMANENTLY LOCKED WORKFLOW RULES
======================================================================

These rules were repeatedly stated by the user and must remain locked for all future Novalyte work.

2.1 Terminal-first workflow

All coding, patching, verification, local testing, Git operations, and cleanup must be performed through terminal commands or a generated terminal updater script. This applies to small and large changes.

Do not give only manual copy/paste file replacements when a terminal process can apply the update safely.

For complicated changes, generate one downloadable PowerShell updater script and give one short command to run it. Avoid asking the user to paste hundreds of PowerShell lines directly because previous long pasted scripts caused multiline prompts and parser errors.

2.2 One unified package/process

When several files must change, use one unified patch or updater. Do not create multiple unnecessary ZIP files for one update.

The expected automated flow is:
1. Validate project path and Git branch.
2. Confirm the working tree is safe before modifying files.
3. Create a temporary rollback backup.
4. Apply only the requested changes.
5. Run syntax and content verification.
6. Launch the correct local project in the browser when requested.
7. Commit and push only when requested or when the agreed updater includes push.
8. Verify local HEAD matches origin/main.
9. Remove temporary update files after success.
10. Print a clear verification checklist.

2.3 Documentation before ZIP

Before creating a full-project ZIP, first create or update a detailed main handover file inside:
Novalyte\Documentation

The documentation must exist before the ZIP is created so it is included inside the project archive.

Do not create random update README files, local notes, temporary changelogs, or unnecessary documentation clutter. Only the main detailed project documentation/handover is needed unless the user explicitly asks for something else.

2.4 No unrelated changes

This is the most important implementation rule:
- Do not change anything the user did not ask to change.
- Do not remove anything the user did not ask to remove.
- Do not add anything the user did not ask to add.
- Do not redesign, rename, relocate, compress, expand, or restyle unrelated components.
- When the user says to adjust only one control, change only that control.
- Preserve current data, calculations, buttons, fields, components, and layout outside the exact requested scope.

The user repeatedly summarized this as:
"Nothing more, nothing less."

2.5 Preserve live data and business settings

Unless specifically requested, preserve:
- Existing client-visible rates
- Provider rates
- Recommended states
- Visible/disabled states
- Existing orders
- Order snapshots
- Investments
- Digital products
- Payment statuses
- Order statuses
- Analytics calculations
- Existing client names and contacts

Old orders must keep their saved snapshot prices even when a service price is edited later.

2.6 Mobile is responsive, not a different product

The phone version should contain the same features, data, and controls as desktop. Only the responsive layout should differ.

A Recommend switch on desktop must remain the same switch on mobile. It must not become a plain checkbox.

Admin navigation panels must not mix. Dashboard, Services, Digital Products, Investments, Orders, and Client View must each show only their own content.

2.7 Visual verification is required

String checks and syntax checks alone are not enough. Previous scripts reported PASS while the rendered UI remained broken.

Future work must verify:

======================================================================

3.1 Order statuses

Required Admin order-status choices:
- Active
- Pending
- Processing
- Completed

3.3 Order financial fields

Orders may contain:
- Provider charge/cost
- Client charge/sale
- Revenue/profit
- Receivable depending on payment state

3.4 Client-safe tracking

The client-side order tracker must never expose provider cost or internal notes.

Client-safe fields include:
- Service ID
- User/client name
- Service name
- Date ordered and time
- Status

The order tracking identifier requested by the user is the provider/service ID, such as 1723, not an automatically generated long order reference.

Multiple clients can have orders under the same service ID. The client can further narrow results by entering their name.

Search text requested on the client tracker:
"Search Service ID or Your Name"

Client-side tracking filters are optional tools for narrowing an already visible order list. The base order list should display available client-safe records and counts.

3.5 Client tracker counts

The client order section should show live counts from Admin data for:
- Pending
- Processing
- Completed

The list columns requested were:
ID | USER / CLIENT NAME | SERVICE | DATE ORDERED | STATUS

3.6 Recommended services

Client Services page:
- Recommended must be the first category option.
- Recommended should be selected by default.
- While Recommended is selected, platform filtering must still work and show only recommended services from the selected platform.
- Service rows remain sorted/grouped appropriately by platform.

Admin Services page:

Admin Digital Products must support Create Order with the necessary client-order fields, similar to service orders but only including fields relevant to digital products.

- Digital product selection
- Client name/reference

3.10 Important Before Ordering notice

The client page contains an Important Before Ordering section. The final content should contain eight warnings, grammatically cleaned but preserving the intended policy:

1. Incorrect quantity, link format, or service selection may delay processing.
2. Invalid or unsupported order details may be cancelled automatically by the system.
3. Once an order has started processing, it can no longer be cancelled or refunded.
4. Correct and complete details help the order run more smoothly.
5. An order may be marked completed even when delivery is incomplete; partial refunds are not guaranteed.
6. In rare cases, an order may be marked completed even when no result is visible. The client should contact Novalyte immediately for checking.
7. The profile, page, post, or channel must remain public while the order is processing. Private or restricted settings can freeze, prevent, or cancel the boosting service.
8. Make sure the account or link is not undergoing another process outside the Novalyte platform. If concurrent outside activity is detected or proven, cancellation or refund requests will not be accommodated.

5. MOBILE ADMIN PROBLEMS AND CURRENT LIMITATION
======================================================================

5.1 Cross-device Admin data is not shared

The phone Admin previously showed no records while desktop had records.

Root cause identified:
The current system falls back to browser-local storage when cloud synchronization is disabled. Desktop and phone use different local storage, so orders, investments, payment changes, and other Admin records do not automatically appear on both devices.

This is not fixable by CSS alone.

True cross-device sharing requires configuring the existing cloud/Supabase integration or another shared backend.

Do not claim that mobile data sharing is fixed unless:
- cloud-config.js is enabled with the correct public configuration
- Admin authentication is working
- tables/schema exist
- read/write operations are verified
- desktop-created records appear on phone
- phone-created or edited records appear on desktop

Never expose a Supabase service-role key in client-side code. Only public anon credentials belong in browser code, protected by proper Row Level Security.

8. KNOWN ERRORS, FAILED APPROACHES, AND LESSONS
======================================================================

8.1 PowerShell pasted-script errors

Long scripts pasted directly into PowerShell caused:
- multiline >> prompts
- nested here-string collisions
- parser failures
- missing closing parentheses/braces
- accidental commands such as a stray S

Locked lesson:
For future non-trivial updates, create a downloadable .ps1 file and give one short execution command.

8.2 FINDSTR quoting error

An earlier verifier printed:
FINDSTR: Cannot open Order

The script incorrectly split a search phrase. Static verifier commands must quote patterns correctly.

8.3 Cleanup path errors

An earlier updater finished the Git push but failed during cleanup with:
- The filename, directory name, or volume label syntax is incorrect.
- A stray S was interpreted as a command.

Locked lesson:
Change out of the extracted updater folder before deleting it, and use literal paths.

8.4 CSS patch stacking

Repeated v6.0.22-v6.0.27 overrides were stacked on top of one another. This caused:
- controls losing borders
- oversized action areas
- clipped select fields
- truncated text
- inconsistent button sizing
- regressions after each apparent fix

Locked lesson:
Inspect and clean the exact current rules. Do not keep adding broad !important blocks blindly.

8.5 Source markup vs. post-render observers

Some fixes relied on MutationObserver and text matching after render. These produced PASS results without reliably fixing the UI.

Locked lesson:
Prefer direct changes in the actual render function/markup when the structure must change. Use CSS-only changes only for true styling adjustments.

8.6 Static PASS is not visual PASS

Several updaters verified only that a CSS marker or function name existed. The browser still looked broken.

Locked lesson:
A complete update requires rendered browser verification.

8.7 Browser cache and wrong local server

At different times, local or live pages appeared unchanged because:
- the browser could be using cached CSS/JS
- a Live Server instance could be serving another folder
- the old URL/domain could be opened

Locked lesson:
Verify the server root, current asset query versions, and use a new cache-busting URL.

8.8 Git CRLF warnings

Warnings such as:
LF will be replaced by CRLF the next time Git touches it
were repeatedly shown.

These warnings alone are not fatal. Do not treat them as a failed update unless git diff --check or syntax validation actually fails.

8.9 GitHub Pages delay

A push may complete before GitHub Pages finishes deploying. Distinguish Git push success from live deployment readiness.
