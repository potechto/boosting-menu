LOCKED DEPLOYMENT RULE
- Every small or large update must use a terminal-first workflow.
- Coding, patching, replacement, verification, local browser launch, Git commit, GitHub push, and cleanup are terminal-driven.
- Do not deliver only a complete project ZIP.
- Use individual patch files for small changes and a patch ZIP only when many files are affected.
- Always state the exact destination path of each updated or new file.
- The normal delivery is a patch package plus one terminal updater.

AUTOMATION ADDED
The root-level NOVALYTE_AUTO_UPDATE_v6_0_1.bat now performs the complete Windows workflow:
1. Finds the main project at the sibling path Novalyte/.
2. Finds NOVALYTE_PATCH_v6_0_1_AUTO_TERMINAL_20260711.zip beside the updater.
3. Extracts the patch into a temporary folder.
4. Backs up files that will be replaced into Novalyte/Documentation/Backups/.
5. Copies all patch files into the exact matching paths in the main project.
6. Verifies required files and runs node --check when Node.js is available.
7. Writes Novalyte/Documentation/VERIFICATION_v6_0_1_20260711.txt.
8. Starts a local Python HTTP server on port 5500 when Python is available.
9. Opens the client and admin pages automatically in the PC browser.
10. Runs git add -A, commits changed files, and pushes the active branch to origin.
11. Opens the GitHub Pages phone-test URL after a successful push.
12. Deletes the downloaded patch ZIP, temporary extraction folder, and updater after successful application.
