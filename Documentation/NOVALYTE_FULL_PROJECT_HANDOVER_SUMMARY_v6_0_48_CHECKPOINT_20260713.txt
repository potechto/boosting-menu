2. PERMANENT USER RULES AND LOCKED WORKFLOW
======================================================================

These rules were repeated many times and must be treated as permanent project requirements.

2.1 Nothing more, nothing less

The strongest rule is:
- Do not change anything the user did not ask to change.
- Do not remove anything the user did not ask to remove.
- Do not add anything the user did not ask to add.
- Do not rename, relocate, restyle, redesign, simplify, reorganize, or "improve" unrelated code or UI.
- Preserve all unaffected data, logic, prices, calculations, controls, files, and layouts.

Before every patch, restate the exact requested scope internally and list the exact files that genuinely require modification. Do not add unrelated files to a patch manifest.

2.2 Terminal-first automated process

All meaningful code updates must be applied through a generated terminal updater or a short terminal-driven process. The user does not want long manual file-edit instructions.

The preferred package format is one ZIP containing:
- One readable PowerShell updater.
- One patch payload directory.
- No extra README, no local-update notes, and no unnecessary documentation files.

The updater should:
1. Verify the exact project path.
2. Verify the Git repository and origin URL.
3. Verify branch main.
4. Check for pre-existing staged files and stop if unrelated staged work exists.
5. Preserve unrelated unstaged files.
6. Verify patch hashes.
7. Back up only the exact target files.
8. Apply only the exact approved target files.
9. Run syntax, content, and Git whitespace checks.
10. Perform a hidden local server probe when a browser change is involved.
11. Commit only approved files.
12. Push to origin/main.
13. Fetch and verify local HEAD equals origin/main.
14. Open only the correct new cache-busted URL when requested.
15. Delete the downloaded patch ZIP, extracted updater, and temporary backup only after verified success.
16. Print a clear final verification checklist.

Use a unique temporary folder containing a GUID. Do not reuse a fixed temp directory because v6.0.35 failed when an old updater file was still locked.

2.3 Hidden processes and correct browser URL

When starting a local Python server, launch it with a hidden window. Do not allow PowerShell or the server window to keep popping up.

Do not launch an old domain, old port, old route, or old cache version. The user previously experienced the old site opening even after a new update.

Use a fresh cache-busting query string after each deployment.

2.4 Documentation before full project ZIP

Before creating a full-project archive:
- Create or update the main handover inside the project's Documentation folder first.
- Then create the full project ZIP so the new handover is inside the archive.

7.1 Do not reuse temporary extraction paths

7.2 Commit message must be a single argument

Generated text files should end with one newline, not an empty extra line.
